## 1. 测试前准备

### 1.1 系统要求

| 项 | 要求 |
| --- | --- |
| 操作系统 | Windows 10/11、macOS、Linux 任一 |
| Node.js | ≥ 18 LTS |
| 包管理 | pnpm ≥ 8 / npm ≥ 9 |
| 浏览器 | Chrome / Edge ≥ 120（推荐），Firefox ≥ 120 也可 |
| 屏幕 | ≥ 1280×720（仪表盘卷的是密度，宽屏更佳） |

### 1.2 启动后端 MetricForge

```bash
# 在后端项目目录
mvn spring-boot:run        # 或 ./gradlew bootRun
```

确认 `http://localhost:8080/api/v1/system/ping` 返回 `{"code":"0",...}`。

### 1.3 启动前端

```bash
cd metriscope-web
pnpm install
pnpm dev                   # 监听 http://localhost:5173
```

浏览器访问 `http://localhost:5173/#/projects`，看到侧栏 + 顶栏即可。

### 1.4 测试样例数据准备

确认以下路径在**后端机器**上存在（默认在仓库根的 `MetriScope/sample-data/` 下）：

| 用途 | 路径 |
| --- | --- |
| Java 源码 | `F:/Desktop/大三下/软件度量应用/项目设计/MetriScope/sample-data/phase1-java-demo/src/main/java` |
| 类图 | `F:/.../sample-data/phase1-java-demo/diagram/class-diagram.puml` |
| 用例图 | `F:/.../sample-data/phase1-java-demo/diagram/use-case-diagram.puml` |
| 活动图 | `F:/.../sample-data/phase1-java-demo/diagram/activity-diagram.puml` |

> 上述路径已硬编码到「上传源码 / 上传图」对话框的快捷按钮，点击即可填入。

### 1.5 浏览器准备

1. 开 DevTools (F12)，切到 **Console** 监控报错
2. **Network** 面板检查 `/api/v1/...` 全部返回 200 + `code: "0"`
3. 清掉旧 localStorage（避免上一次的 baseUrl 残留）：
   ```js
   localStorage.removeItem("metriscope")
   ```

## 14. 附录：常用排错

| 现象 | 可能原因 | 处理 |
| --- | --- | --- |
| 心跳指示灯一直红 | 后端没启动 / 端口不是 8080 | 先 `curl http://localhost:8080/api/v1/system/ping` 自测 |
| 跨域报错 | dev proxy 没生效 | 检查 `vite.config.ts`，确认重启过 dev server |
| 上传源码报 SOURCE_PATH_INVALID | 路径在前端机但不在后端机 | 后端读的是后端进程的本地文件系统，路径必须后端可见 |
| `/metrics` 一直空状态 | 项目未分析 / 上一次分析失败 | 回 /analysis 看任务状态 |
| Typst-PDF 失败但回退 | 本机没装 typst CLI | exportType = `PDF_TYPST_FALLBACK`，正常现象 |
| 控制台 React Hook 警告 | 应该没有 | 立即贴给开发，是 bug |
| localStorage 卡住老 baseUrl | 之前测过 Settings 页 | F12 Application → Local Storage → 删 `metriscope` |
