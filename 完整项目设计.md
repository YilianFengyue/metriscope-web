# 一、项目定义

## 1.1 项目名称

**面向 Java 项目的软件度量、质量评估与智能优化平台**

## 1.2 项目一句话介绍

用户导入 **Java 源码项目** 和可选的 **PlantUML 设计图**，系统自动完成 **代码解析、度量计算、风险识别、历史对比、报告导出**；并可通过 **MCP** 将分析能力暴露给 Claude Code / Codex 等 Coding Agent，形成“分析—优化—复测”的闭环。

## 1.3 项目价值

这个项目不是单纯“统计代码行数”，而是做一个完整的软件度量平台，解决四类问题：

1. 代码质量无法量化，只能凭经验判断。
2. 设计图和代码脱节，难以做一致性检查。
3. 改进前后缺少统一对比依据。
4. 度量结果只能看，不能继续驱动 AI 优化。

## 1.4 目标用户

有三类：

### A. 课程学习者

需要完成实验、生成报告、展示图表、对比前后变化。

### B. 开发者

需要发现高风险类、高复杂度方法、高耦合模块。

### C. 项目管理者/组长

需要快速看到整体质量趋势、风险热点、估算结果。

# 二、项目总功能列表

我建议把功能拆成 8 个一级模块。

## 模块 M1：项目导入与输入管理

负责接收分析对象和项目参数。

### M1-1 Java 源码导入

**做什么：**
支持用户选择本地目录或上传压缩包，递归扫描 Java 文件。

**输入：**

* 本地项目目录路径
* zip 文件

**输出：**

* 项目基础信息
* 文件清单
* 可解析文件数
* 解析任务 ID

### M1-2 PlantUML 设计图导入

**做什么：**
支持导入 `.puml` 类图、用例图、活动图，用于设计阶段度量和一致性检查。

**输入：**

* `.puml` 文件

**输出：**

* 图类型
* 图中元素列表
* 关系列表
* 解析状态

### M1-3 项目元数据录入

**做什么：**
录入项目管理相关参数，用于用例点、功能点、COCOMO、成本估算。

**输入：**

* 项目名称
* 项目说明
* 人员数量
* 人周成本
* 项目起止时间
* 环境因子
* 技术复杂度因子

**输出：**

* 项目元数据对象

### M1-4 导入记录管理

**做什么：**
记录每一次导入操作，方便后续做快照和历史对比。

**输入：**

* 项目 ID
* 导入类型
* 时间
* 用户参数

**输出：**

* 导入历史记录

---

## 模块 M2：解析与中间模型构建

负责把源码和设计图转成可计算结构。

### M2-1 Java AST 解析

**做什么：**
使用 JavaParser 解析 Java 源文件，得到 AST。

**输入：**

* Java 文件集合

**输出：**

* CompilationUnit 集合
* 解析错误列表

### M2-2 代码结构抽取

**做什么：**
从 AST 抽取项目结构信息。

**输出内容：**

* 包
* 类 / 接口 / 枚举
* 字段
* 方法
* 继承关系
* 实现关系
* 方法参数
* 返回值类型
* 注释信息
* 源码位置（文件、起止行）

### M2-3 依赖关系抽取

**做什么：**
抽取类与类之间的引用、调用、关联、聚合、继承等关系。

**输出：**

* 类依赖边
* 方法调用边
* 包依赖边

### M2-4 PlantUML 结构抽取

**做什么：**
解析类图/用例图/活动图中的元素和关系。

**输出：**

* 类图实体
* 类图关系
* 用例
* 参与者
* 活动节点

### M2-5 统一中间模型

**做什么：**
把 Java 结构和设计图结构转成系统内部统一模型，供后续计算。

**输出：**

* ProjectModel
* PackageModel
* ClassModel
* MethodModel
* DiagramModel

## 模块 M3：核心度量引擎

这是项目最核心的模块。

### M3-1 代码规模度量

**做什么：**
统计代码规模相关指标。

**指标：**

* 物理行数
* 逻辑行数
* 注释行数
* 空白行数
* 类数
* 方法数
* 包数
* 字段数

**输出：**

* 项目级规模结果
* 类级规模结果
* 文件级规模结果

### M3-2 圈复杂度度量

**做什么：**
计算方法级和类级 McCabe 圈复杂度。

**指标：**

* 方法圈复杂度
* 类平均圈复杂度
* 类最大圈复杂度
* 项目平均复杂度

**输出：**

* 方法复杂度清单
* 高复杂度方法排行

### M3-3 CK 度量集

**做什么：**
计算经典 CK 六项。

**指标：**

* WMC
* RFC
* NOC
* DIT
* CBO
* LCOM

**输出：**

* 每个类的 CK 指标
* 项目级 CK 汇总

### M3-4 LK 度量集

**做什么：**
计算课程要求的 LK 指标。

**建议实现：**

* CS
* NOO
* NOA
* SI

**输出：**

* 每个类的 LK 指标
* 项目级 LK 汇总

### M3-5 设计阶段估算

**做什么：**
基于设计图和手工输入做估算。

**子功能：**

* 用例点计算
* 功能点计算
* 特征点计算（可后置）
* 角色权重计算
* 技术复杂度调整
* 环境复杂度调整

**输出：**

* 用例点结果
* 功能点结果
* 估算中间过程

### M3-6 工作量与成本估算

**做什么：**
根据规模和复杂度做工时/成本估计。

**建议：**

* COCOMO 基础版
* 人力成本估计
* 周期估计

**输出：**

* 预计工作量
* 预计开发成本
* 推荐周期范围

## 模块 M4：规则分析与风险识别

负责把“指标”转成“能解释的问题”。

### M4-1 阈值规则配置

**做什么：**
为各指标配置阈值。

**示例：**

* 圈复杂度 > 10 视为高风险
* CBO > 8 视为高耦合
* LCOM 高于阈值视为低内聚

**输出：**

* 阈值配置项

### M4-2 风险识别

**做什么：**
根据阈值识别高风险对象。

**风险类型：**

* 高复杂度方法
* 高耦合类
* 低内聚类
* 过深继承类
* 职责过重类

**输出：**

* 风险对象列表
* 风险原因说明
* 风险等级

### M4-3 类图与代码一致性检查

**做什么：**
检查设计图和代码是否一致。

**建议第一版只做类图：**

* 类名是否存在
* 继承关系是否匹配
* 实现关系是否匹配
* 核心关联是否匹配

**输出：**

* 一致项
* 缺失项
* 不匹配项

### M4-4 改进建议生成

**做什么：**
基于规则生成结构化建议。

**示例：**

* 某类耦合过高，建议拆分职责
* 某方法复杂度过高，建议提取子方法
* 某类 RFC 过高，建议减少公共接口扩散

**输出：**

* 建议文本
* 建议优先级
* 建议适用对象

## 3. 后端项目结构

建议直接用单体分层，不要一上来微服务。

```text
metric-platform/
  metric-platform-server/
    src/main/java/org/csu/
      common/
        api/
        constant/
        enums/
        exception/
        util/
      config/
      infrastructure/
        parser/
          java/
          plantuml/
        storage/
        report/
        mcp/
      modules/
        project/
          controller/
          service/
          service/impl/
          mapper/
          entity/
          dto/
          vo/
          convert/
        source/
        diagram/
        analysis/
        snapshot/
        metric/

        export/
      MetricPlatformApplication.java
  metric-platform-web/
```

### 包职责

- `common`：统一响应、异常、常量、枚举、工具类
- `config`：MyBatis-Plus、MVC、跨域、Jackson、文件上传配置
- `infrastructure.parser`：JavaParser 和 PlantUML 解析适配
- `infrastructure.report`：Typst 模板渲染和 PDF 生成
- `infrastructure.mcp`：MCP tools 暴露层
- `modules.*`：真正的业务模块

### 关键约束

- `controller` 只收发 DTO/VO，不写业务
- `service` 写业务编排
- `mapper` 只做持久化
- 复杂度量计算放在 `metric` 模块，不要散落到 controller/service
- MCP 不允许直接查数据库，必须调用 service

```json

  "list": [],
```

### 4.3 错误码建议

- `0` 成功
- `4001` 参数错误
- `4002` 文件格式错误
- `4003` 解析失败
- `4041` 项目不存在
- `4042` 快照不存在
- `5001` 系统异常
- `5002` 报告生成失败

## 8. MyBatis-Plus 使用规约

这个也要提前定。

### 固定规则

1. **单表 CRUD**：全部走 MyBatis-Plus `BaseMapper` / `IService`
2. **复杂统计 SQL**：允许 XML
3. **分页**：统一走 MyBatis-Plus 分页插件
4. **代码生成**：只用于首轮脚手架，不允许后期反复覆盖手写业务代码

MyBatis-Plus 官方文档明确提供了持久层接口方法和代码生成配置，这很适合你们先把 entity / mapper / service 骨架批量生成，再手写复杂业务。([MyBatis-Plus](https://baomidou.com/en/guides/data-interface/?utm_source=chatgpt.com))

### 推荐做法

- `entity`、`mapper`、`service`、`serviceImpl` 初版可生成
- `controller` 建议手写
- 复杂查询统一放 `xxxMapper.xml`
- 所有表统一：
  - `id`
  - `created_at`
  - `updated_at`
  - `deleted`
